End-to-End: How a Query Travels
from Grafana Plugin to AI Answer

Every user query takes a precise, governed path through the platform — from the Grafana panel through LiteLLM's unified LLM gateway to the observability stack and back as a streamed RCA.

1
User — Grafana Plugin Panel or React UI
Entry Point · User Interaction Layer
An on-call SRE types a natural language question in the Grafana Plugin Panel ("investigate high error rate on checkout") or the React web app chat interface. The Grafana plugin is a standard Grafana app plugin that embeds an iframe with a WebSocket connection to the backend. A REST request or WebSocket message is initiated.
Grafana App Plugin React UI (Vite + shadcn/ui) WebSocket (native) REST API

Auth · Session Management · Request Routing
The FastAPI backend receives the request on /ws/agent/{session_id} (WebSocket) or /api/v1/incidents/{id}/investigate (REST). JWT authentication is validated, the incident context is loaded from PostgreSQL, and a new Investigation session is created. The session ID is used to fan out streaming results to the correct WebSocket subscribers via Redis pub/sub. The backend then dispatches the LangGraph investigation graph as a background task.

❓ FAQ

Frequently Asked Questions

Everything engineering and leadership need to know before deploying the platform.

🔒
Security & Data Privacy
Does the platform write anything to Grafana, Mimir, Loki, or Kubernetes?▼

No. The platform is strictly read-only. All MCP tools query APIs for data — they never write, modify, scale, restart, or delete any resources. The only writes are to the platform's own PostgreSQL database (incidents, investigations, SAAFE scores) and optional cosmetic Grafana annotations (incident markers on dashboards).

Does any production data leave our infrastructure?▼

Only summarized, curated payloads are sent to the LLM. Alert names, metric values, log pattern summaries, and flame graph top-function lists are sent for AI analysis. Full raw log lines and complete trace payloads are never sent. Use Azure OpenAI (your own tenant) or a self-hosted model to keep all data within your cloud boundary — the LLM backend is swappable via OPENAI_BASE_URL.

How is PII / PCI / PHI data handled before LLM analysis?▼

The MCP ingestion layer runs every log snippet and annotation through a configurable redaction pipeline before any data reaches the LLM context. PCI patterns (card numbers, CVV heuristics), PHI stream labels (sensitivity=phi), and PII fields (emails, IPs, SSN patterns) are either blocked or replaced with [REDACTED] tokens. The pipeline is defined in the MCP Server YAML config and is mandatory — it cannot be disabled.

What authentication and authorization model does the platform use?▼

The FastAPI backend enforces JWT-based authentication on all endpoints. Role-based access control (RBAC) maps users to one of five roles: admin, editor, viewer, ai-agent, and read-only. Service accounts used by the AI agents have the narrowest possible scope — read-only access to specific datasources only. All API keys are stored as Kubernetes secrets or environment variables — never in source code.

Are all AI agent actions logged for audit purposes?▼

Yes. Every MCP tool call (query sent, parameters, duration, token count, user/agent identity) is written to a Loki audit stream tagged {app="mcp-server", env="prod"}. These logs are immutable, queryable via LogQL, and retained according to your Loki retention policy. The audit stream cannot be disabled — it is a core platform invariant.

🧠
AI & Intelligence
How long does a full RCA investigation take?▼

Typically 45–90 seconds for a complete investigation across all signal sources (metrics, logs, traces, profiles, frontend). The Planner Agent dispatches specialist agents in parallel — you see partial findings stream in via WebSocket within 10–15 seconds of triggering. The Synthesizer writes the final RCA narrative once all agents report back. Simple single-signal queries (e.g. "what's the error rate for checkout?") complete in under 5 seconds.

How accurate is the RCA? Can the AI be wrong?▼

The Synthesizer always includes a confidence score (0–100) and labels each piece of evidence as High / Medium / Low confidence. It is designed to say "insufficient data" rather than hallucinate a root cause. The system never guesses — it only states what the observed data shows. We recommend treating all RCA outputs as a strong starting hypothesis that an engineer confirms, not an autonomous decision.

How is the SAAFE score calculated?▼

SAAFE is a weighted composite of five normalized sub-scores (0–100 each): Security (CVE exposure + TLS certificate age), Availability (error rate + SLO burn rate), Anomaly (rolling Z-score magnitude across RED metrics), Failure (pod restarts + OOMKilled events), and Events (deployment risk in the last 2h). Weights are configurable per team in saafe_weights.yaml. The score refreshes every 5 minutes via a Celery background task.

Can I use a different LLM provider instead of OpenAI?▼

Yes. The LLM backend is abstracted via LangChain's BaseChatModel interface. Set OPENAI_BASE_URL to point to any OpenAI-compatible endpoint: Azure OpenAI, Ollama (local LLaMA3, Mistral), Groq, Anthropic Claude via a proxy, or Amazon Bedrock. The planner and synthesizer agents use gpt-4o class models by default; fast sub-tasks use gpt-4o-mini class models. Both tiers are configurable independently.

How does the RAPTOR Knowledge Base improve over time?▼

Every resolved investigation is automatically ingested into the RAPTOR knowledge base as an institutional memory entry, tagged with the affected services, severity, and extracted RCA pattern. The next time a similar incident occurs, the Planner Agent retrieves the most relevant past RCAs as context before generating its investigation plan. This creates a self-learning feedback loop — the platform gets smarter with every incident it resolves.

🔔
Alert Intelligence & Correlation
Can the Alert Clustering engine handle bursty alert storms?▼

Yes. Incoming alerts are buffered in Redis Streams before clustering, absorbing sudden bursts of thousands of alerts without event loss. The streaming HDBSCAN implementation processes alerts in micro-batches with a configurable flush interval (default 500ms). In load tests the engine sustained 10,000 alerts/minute while maintaining sub-1-second clustering latency and less than 200MB memory usage.

How does the 3-layer correlation engine work?▼

Layer 1 (Temporal): alerts firing within a configurable time window (default 5 minutes) are grouped together. Layer 2 (Topological): groups whose services have a dependency relationship in the service graph are merged — a DB slowdown alert and a checkout error alert become one incident. Layer 3 (Semantic): remaining groups with semantically similar alert text (cosine similarity > 0.75 on OpenAI embeddings) are merged. The three layers together typically reduce alert noise by 85–95%.

What happens when an alert is suppressed by the correlation engine?▼

Suppressed alerts are not deleted — they are linked to their parent alert group and stored in PostgreSQL with a status=suppressed flag. The UI shows a "12 suppressed" badge on the parent alert cluster, and engineers can expand the group to see all contributing alerts. Suppressed alerts still appear in the audit log and can be re-evaluated if the parent group is dismissed.

How is the P1 / P2 severity automatically determined?▼

The platform combines three signals to auto-classify severity: (1) the Alertmanager severity label on incoming alerts, (2) the SAAFE score of affected services (SAAFE < 40 forces P1), and (3) a correlation group size heuristic — groups with 5+ correlated firing alerts are escalated to at least P2. The final severity is the maximum of these three signals. Teams can override via the UI or API at any time.

Integration & Deployment
What is the minimum infrastructure needed to run this?▼

The backend requires 2 vCPUs / 4 GB RAM minimum (4 vCPUs / 8 GB recommended). You need an existing Grafana stack with at minimum Mimir and Loki. Tempo is strongly recommended for trace correlation. PostgreSQL 16 and Redis 7 are required. ChromaDB is optional — only needed for the RAPTOR knowledge base. The entire stack can be stood up locally with make up via Docker Compose in under 3 minutes.

Does the platform support multiple Kubernetes clusters?▼

Yes. Multi-cluster support is native via Mimir and Loki's tenant/cluster label selectors. Each cluster is identified by a cluster label in metric and log streams. SAAFE scores, SLO dashboards, and alert correlation all support filtering and aggregation by cluster. The service topology graph supports cross-cluster service edges using Tempo distributed tracing span data.

How do I connect the platform to an existing Alertmanager?▼

Add a webhook receiver to your Alertmanager config pointing to https://<your-platform>/api/v1/alerts/webhook/alertmanager. The endpoint accepts the standard Alertmanager v2 webhook payload. It acknowledges in under 200ms and processes alerts asynchronously in the background. No changes to your existing alert routing rules are needed — the platform is additive.

Does the Faro RUM integration require changes to our frontend apps?▼

Yes, a small one-time SDK integration. Add the Grafana Faro Web SDK to your frontend app (~5KB gzipped), configure the collector endpoint, and add the beforeSend hook for PII stripping (a 10-line snippet). After that, Web Vitals (LCP, INP, CLS, TTFB), JavaScript errors, and user session counts flow automatically into the platform without any further changes.

Can I run this without Pyroscope?▼

Yes. All signal sources are independently optional. If Pyroscope is not configured, the Profiles Agent gracefully skips and the RCA report notes "profiling data unavailable." The platform degrades gracefully for any missing signal source — the Planner Agent only dispatches agents for signals that are configured and healthy according to the /health endpoint.

Operations & Cost
What is the typical OpenAI API cost per investigation?▼

A full 6-agent investigation (all signals) typically consumes 3,000–6,000 input tokens and 500–800 output tokens using GPT-4o. At current pricing this is approximately $0.05–$0.12 per investigation. Fast sub-tasks (log summarization, label lookup) use GPT-4o-mini at ~10× lower cost. For a team handling 50 P1/P2 incidents per month the monthly AI cost is typically under $20. Token budgets per team are enforced via the governance layer.

How do I add runbooks and post-mortems to the Knowledge Base?▼

Use the /api/v1/knowledge/ingest endpoint or the KB Upload UI in the Knowledge page. Supported formats: Markdown, PDF, plain text, and Confluence REST API links. The RAPTOR pipeline automatically chunks, embeds, clusters, and builds the hierarchical summary tree. Ingestion of a 20-page runbook takes approximately 30–45 seconds. Resolved incidents are auto-ingested without any manual step.

How do I configure SLO targets for each service?▼

SLO targets are defined per service in the Service Registry — either via the Settings UI or the /api/v1/services API. Each service record holds a slo_target (e.g. 99.9) and an optional error_budget_policy. The Metrics Agent automatically computes the 1h and 5m burn rates against the configured target during every investigation. No PromQL rule changes are needed — all SLO math happens inside the agent.

What monitoring exists for the platform itself?▼

The platform is fully self-instrumented. The FastAPI backend exposes Prometheus metrics at /metrics (prefixed novasre_) covering: investigation duration, tool call latency per signal source, alert correlation throughput, WebSocket connection count, and LLM token usage. These metrics can be scraped by your existing Mimir instance and displayed on the included self-monitoring Grafana dashboard. Structured logs stream to Loki and traces are sent to Tempo via OpenTelemetry.
